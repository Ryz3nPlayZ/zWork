# Pre-v1 release plan (macOS + Windows + Linux)

## Problem
zWork already ships cross-platform builds, but pre-v1 quality is not yet guaranteed: key user flows must be consistently reliable, UX must feel smooth (no noticeable lag), and release/install behavior must be stable across all three OSes.

## Proposed approach
Run a release-hardening plan in 8 tracks: define an explicit quality bar, lock critical flows, eliminate performance hotspots, harden OS-specific behavior, strengthen packaging/updater reliability, expand automated and manual regression coverage, close high-impact defects, and ship through a release-candidate gate with rollback readiness.

## Todos
1. **release-quality-bar**  
   Define pre-v1 ship criteria: critical flow pass requirements, crash-free target, latency budgets (chat response, UI interactions, startup), and per-OS release gates.
2. **critical-flow-inventory**  
   Convert intended functionality into a canonical checklist of must-pass flows (file operations, browser/tab summarize, artifact creation/editing, approvals/safety prompts, workflow save/reuse, update flow).
3. **gap-audit-current-state**  
   Execute baseline audit against current app using BENCHMARKS + targeted exploratory checks; classify failures by severity and OS.
4. **fix-p0-p1-functionality**  
   Fix all P0/P1 bugs found in critical flows, including data-loss, stuck-run, incorrect plan/action, and permission/safety regressions.
5. **perf-budget-enforcement**  
   Add and enforce performance checks: startup time, first-action latency, render/input jank, and long-task progress responsiveness; remove major bottlenecks.
6. **cross-platform-hardening**  
   Resolve OS-specific issues in paths, permissions, process management, browser integration, packaging runtime dependencies, and updater behavior.
7. **release-pipeline-hardening**  
   Validate and harden build/package/signing/release scripts and GitHub Actions outputs (artifacts, signatures, latest.json, installer usability).
8. **automated-regression-expansion**  
   Expand backend/frontend automated tests for critical flows and OS-sensitive code paths; ensure failures block release.
9. **manual-smoke-matrix**  
   Run manual smoke matrix on macOS/Windows/Linux for install, first-run onboarding, core task completion, update check, and uninstall/reinstall sanity.
10. **docs-and-support-readiness**  
   Finalize user-facing docs for install/troubleshooting per OS and internal runbooks for release/recovery.
11. **rc-freeze-and-triage**  
   Cut release candidate, freeze scope to bug/perf fixes only, triage late findings daily, and track explicit go/no-go criteria.
12. **pre-v1-ship-and-post-ship-watch**  
   Publish pre-v1 release, monitor crash/error/perf telemetry and user reports closely, and ship fast follow-up patch window for regressions.

## Notes / considerations
- Existing assets to reuse: `BENCHMARKS.md`, `docs/RELEASES.md`, release workflow, packaging scripts, current backend tests.
- Cross-platform pass means behavior parity, not just successful builds.
- “No lag” should be treated as measurable budgets, not subjective feel only.
- Release gate should require both automated pass and human smoke pass on all three OSes.
